# Implementation Plan: Async Responses (Background Mode)

**Branch**: `044-async-responses` | **Date**: 2026-03-05 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/044-async-responses/spec.md`

## Summary

Add `background: true` support to the Responses API, enabling fire-and-forget inference requests for long-running agentic workloads. The server accepts background requests immediately (returning status `queued`), processes them asynchronously via a distributed worker architecture, and lets clients poll for results. The same binary runs in three modes (`gateway`, `worker`, `integrated`) controlled by a `--mode` flag.

Key design decisions from research:
- Extend `ResponseStore` with `UpdateResponse` and `ClaimQueuedResponse` methods (R-001, R-002)
- Persist the full `CreateResponseRequest` alongside the response for worker reconstruction (R-004)
- Worker heartbeat on response records for stale detection, no separate worker registry (R-005)
- TTL cleanup piggybacks on the worker poll cycle (R-006)
- Distributed cancellation via status polling, in-process cancellation via context registry for integrated mode (R-007)

## Technical Context

**Language/Version**: Go 1.22+ (consistent with all existing specs)
**Primary Dependencies**: Go standard library only for core. `pgx/v5` for PostgreSQL adapter (existing dependency).
**Storage**: PostgreSQL 14+ for production (distributed mode). In-memory for development/testing (integrated mode).
**Testing**: Go `testing` package. Integration tests with subprocess server binary.
**Target Platform**: Linux containers on Kubernetes
**Project Type**: Web service (inference gateway)
**Performance Goals**: Background request acceptance < 1s. Worker poll interval 5s (configurable). No performance regression for synchronous requests.
**Constraints**: Gateway and worker must be independently scalable. No external dependencies beyond PostgreSQL (already present).
**Scale/Scope**: Hundreds of concurrent background requests across multiple worker pods.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Interface-First Design | Pass | New storage methods defined as interface extensions. Worker defined as interface for testability. |
| II. Zero External Dependencies (Core) | Pass | No new external dependencies. PostgreSQL adapter extends existing `pgx/v5` usage. |
| III. Nil-Safe Composition | Pass | Background processing is optional (nil worker in gateway mode). Integrated mode composes gateway + worker. |
| IV. Typed Error Domain | Pass | Background failures use existing `APIError` type. New `failed` responses carry structured error. |
| V. Validate Early, Fail Fast | Pass | `background: true` + `store: false` and `background: true` + `stream: true` validated before processing. |
| VI. Protocol-Agnostic Provider | Pass | Workers invoke the engine, which calls providers through the existing interface. No provider changes. |
| VII. Streaming as First-Class | Pass | Background mode and streaming are mutually exclusive. No streaming changes needed. |
| VIII. Context Carries Cross-Cutting | Pass | Worker propagates identity/tenant from the original request context. Cancellation via context. |
| IX. Kubernetes-Native | Pass | Gateway and worker run as separate Deployments. Quickstart demonstrates the pattern. |
| Documentation Standards | Pass | New deployment pattern requires quickstart. API changes require reference docs. Config changes require config-reference updates. |

**Post-design re-check**: All principles still pass. The `--mode` flag is a standard deployment configuration, not a new external dependency. Storage interface extensions follow Interface-First Design.
