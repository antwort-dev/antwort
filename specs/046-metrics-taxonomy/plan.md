# Implementation Plan: Production Metrics Taxonomy

**Branch**: `046-metrics-taxonomy` | **Date**: 2026-03-29 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/046-metrics-taxonomy/spec.md`

## Summary

Expand Antwort's Prometheus metrics from the 12 core metrics (spec 013) to a comprehensive 7-layer taxonomy covering Responses API, Engine/Agentic Loop, Storage, Files/Vector Store, and Background Workers. This adds ~23 new metrics across these layers, giving operators complete visibility into every subsystem. The existing `pkg/observability/` package and `prometheus/client_golang` dependency are reused.

## Technical Context

**Language/Version**: Go 1.22+ (consistent with all existing specs)
**Primary Dependencies**: `prometheus/client_golang` (existing from spec 013). No new dependencies.
**Storage**: N/A (metrics are ephemeral Prometheus scrape targets, no persistence)
**Testing**: Go standard `testing` package + `prometheus/client_golang/testutil` for metric assertions
**Target Platform**: Linux server (same as existing Antwort deployment)
**Project Type**: Web service (gateway)
**Performance Goals**: < 1ms overhead per request (SC-004). Prometheus counter/histogram ops are lock-free atomics.
**Constraints**: No new external dependencies (constitution Principle II). No feature flags for metrics.
**Scale/Scope**: 23 new metrics (35 total), 6 packages instrumented, ~15 files modified

## Phase 0: Outline & Research

### Research Findings

The codebase has an existing observability foundation from spec 013:

**Existing Infrastructure**:
- `pkg/observability/metrics.go`: Contains 12 metrics across 3 areas (HTTP, Provider, Tools, Rate Limiting, OTel GenAI)
- Metrics are registered in `init()` function using `prometheus.MustRegister()`
- LLM-tuned bucket configuration: `LLMBuckets = []float64{0.1, 0.5, 1, 2, 5, 10, 30, 60, 120}`
- `RecordGenAIMetrics()` helper function for bulk OTel gen_ai.* metric recording
- HTTP middleware in `pkg/observability/middleware.go` captures request-level metrics
- Metrics exposed at `/metrics` endpoint via `promhttp.Handler()` in `cmd/server/main.go`

**Existing Metric Recording Points**:
- **HTTP Layer** (`pkg/observability/middleware.go`): Captures request duration and streaming connections
- **Engine/Provider** (`pkg/engine/loop.go`): Records provider latency, token counts, and tool executions
- Tool executions recorded in `executeToolsConcurrently()`, `executeToolsSequentially()`, and `executeToolsWithEvents()` with tool name and status labels

**Storage Interface** (`pkg/storage/`):
- `transport.ResponseStore` interface for response persistence (memory, postgres implementations)
- Memory store at `pkg/storage/memory/store.go`
- PostgreSQL store at `pkg/storage/postgres/store.go` with connection pooling
- No existing storage metrics; instrumentation requires wrapper pattern

**Background Workers** (`pkg/engine/background.go`):
- `Worker` struct manages polling, heartbeats, and stale detection
- `pollOnce()` method claims responses from queue
- No existing background metrics; needs direct instrumentation

**Files and Vector Store** (`pkg/files/`, `pkg/tools/builtins/filesearch/`):
- `pkg/files/api.go`: Handler for file upload/listing
- `pkg/files/pipeline.go`: Ingestion pipeline orchestration
- `pkg/tools/builtins/filesearch/api.go`: Vector store search API
- `pkg/tools/builtins/filesearch/provider.go`: Backend vector store implementation
- No existing file/search metrics; needs direct instrumentation

### Research Questions Resolved

All technical unknowns are clarified:
- **Metric registration pattern**: `prometheus.NewCounterVec()`, `prometheus.NewHistogramVec()`, `prometheus.NewGauge()` → `prometheus.MustRegister()` in `init()`
- **Label cardinality**: No relabeling in code; operators handle via Prometheus configuration
- **Instrumentation style**: Direct metric recording (e.g., `metric.WithLabelValues().Inc()`) at observation points
- **Histogram buckets**: Reuse existing `LLMBuckets` for durations; define new buckets if needed (e.g., token counts)
- **Metrics endpoint**: Already exposed at `/metrics` (configurable path); no changes needed to server setup

### Contracts: Metric Interface Specification

(See `contracts/metrics-contract.md`)

**Metric Registration Contract**:
- All metrics MUST be registered in `pkg/observability/init()` using `prometheus.MustRegister()`
- Metric names follow Prometheus convention: `antwort_` prefix, unit in name (`_seconds`, `_bytes`), `_total` suffix for counters
- All histograms MUST use `LLMBuckets` for durations or `TokenBuckets` for token counts
- No dynamic metric creation; all metrics are static and registered at startup

**Recording Contract**:
- Counters: Use `.WithLabelValues(...).Inc()` or `.Add(float64(value))`
- Histograms: Use `.WithLabelValues(...).Observe(value)`
- Gauges: Use `.Set(value)`, `.Inc()`, `.Dec()`
- Recording MUST occur synchronously at observation points (no async batching)
- Overhead MUST be <1μs per recording (atomic operations only)

### Implementation Approach

**Step 1: Register new metrics in `pkg/observability/metrics.go`**
- Add 23 new metric definitions (CounterVec, HistogramVec, Gauge)
- Use existing `LLMBuckets` for all duration histograms
- Register all in `init()` function

**Step 2: Instrument each layer**
- **Engine** (`pkg/engine/loop.go`): Add iteration counting, max-iterations-hit, tool duration, conversation depth
- **Storage** (`pkg/storage/{memory,postgres}/store.go`): Wrap operations with duration recording; track active responses and connections
- **Files** (`pkg/files/api.go`, `pkg/files/pipeline.go`): Add upload counter and ingestion duration
- **Vector Store** (`pkg/tools/builtins/filesearch/api.go`, `provider.go`): Add search counter and duration
- **Background** (`pkg/engine/background.go`): Add queue depth, claim counter, stale detection, heartbeat age

**Step 3: Test**
- Unit tests for metric registration (existing pattern in `pkg/observability/metrics_test.go`)
- Integration tests for metric recording (new tests in each package's test files)
- No changes to spec 013 metrics or middleware

**Step 4: Documentation**
- Update config reference to document all metrics and label cardinality risks
- Add Grafana dashboard example (reference only; operators build their own)

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| High cardinality labels (model, tool_name) | Document in config reference; operators use Prometheus relabeling. No code-level limit. |
| Metric recording overhead | Use atomic operations only; no locks or allocations. Testing will verify <1μs overhead. |
| Storage instrumentation complexity | Use wrapper/decorator pattern to wrap existing store implementations. No interface changes. |
| Backwards compatibility | All changes are additive. Spec 013 metrics unchanged. |
