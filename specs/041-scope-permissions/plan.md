# Implementation Plan: Scope-Based Authorization and Resource Permissions

**Branch**: `041-scope-permissions` | **Date**: 2026-03-04 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/041-scope-permissions/spec.md`

## Summary

Add scope-based endpoint authorization with configurable role-to-scope mapping, three-level resource permissions (`owner|group|others`) for vector stores and files, and vector store union merge with agent profiles. Scope enforcement is a new middleware that checks effective scopes (JWT + role expansion) against hardcoded endpoint requirements. Permissions enable sharing resources within a tenant (group) or across tenants (others). All features are nil-safe: unconfigured means no enforcement (backward compatible).

## Design Decisions

### Scope Middleware as Separate Package

The scope middleware lives in `pkg/auth/scope/` rather than in the auth middleware. This keeps authentication (who are you?) separate from authorization (what can you do?). The middleware stacking in `cmd/server/main.go` applies scope after auth:

```
Request → CORS → Metrics → Auth → Scope → Handler
```

### Role Expansion Algorithm

At startup:
1. Parse role_scopes config into `map[string][]string`
2. For each role, recursively resolve references (entries that match a role name)
3. Detect cycles using a visited set per resolution chain
4. Flatten into `map[string]map[string]bool` (role -> set of effective scopes)
5. Cache the expanded map for O(1) lookup at request time

### Permission Check Integration

Permission checks extend the existing owner check pattern. The `ownerAllowed`/`vsOwnerAllowed` helpers from Spec 040 are extended to also check group and others permissions when the caller is not the owner and not admin. The check order is:

1. Owner? -> always allowed (owner level is always `rwd`)
2. Admin (same tenant)? -> read + delete allowed (Spec 040)
3. Same tenant (group)? -> check `permissions.Group` for the requested operation
4. Different tenant (others)? -> check `permissions.Others` for the requested operation
5. None of the above? -> deny (404)

### Permissions Input vs Output Format

- **Input** (create/update): JSON object `{"group": "r", "others": "r"}`
- **Output** (GET responses): Compact string `"rwd|r--|r--"`
- Owner level is always `rwd` in the output, regardless of what the input says
