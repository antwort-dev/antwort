# Implementation Plan: Resource Ownership

**Branch**: `040-resource-ownership` | **Date**: 2026-03-04 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/040-resource-ownership/spec.md`

## Summary

Add per-user data isolation to responses, conversations, and vector stores. Every resource gets an `owner` field set from `Identity.Subject` at creation time. Storage queries filter by owner, returning 404 for non-owner access. An admin role (configurable, extracted from JWT claims) overrides ownership filtering for read and delete operations within the admin's own tenant. Backward compatible: no-auth deployments continue to work unchanged.

## Design Decisions

### Owner Context Pattern

Two options were evaluated:

**Option A: Dedicated `SetOwner`/`GetOwner` context helpers** (like `SetTenant`/`GetTenant`)
- Pro: Explicit, clear separation between owner and tenant
- Con: Requires middleware to inject owner separately from identity

**Option B: Extract owner from `Identity.Subject` at query time** (like Files API `userFromCtx`)
- Pro: No additional context injection, identity already in context
- Con: Each store must call `auth.IdentityFromContext(ctx)` directly

**Decision: Option B.** Follow the Files API pattern. The identity is already in context. Each store method extracts the subject when needed. This avoids adding another context key and keeps the approach consistent with the proven `userFromCtx` pattern.

### Admin Role Check

The admin role is checked by reading roles from `Identity.Metadata["roles"]`. The JWT authenticator extracts roles from a configurable claim path (default: `realm_access.roles`) and stores them in metadata as a comma-separated string. A helper function `IsAdmin(identity, adminRoleName)` checks if the role is present.

For API key auth, the admin role can be configured directly in the key's identity metadata.

### Authorization Flow

```
Request arrives
  → Auth middleware: extract Identity (existing)
  → Auth middleware: inject Identity + tenant into context (existing)
  → Handler: calls store method with context
  → Store method:
      1. Get identity from context (auth.IdentityFromContext)
      2. If identity is nil → no filtering (backward compat)
      3. Get tenant_id from context (storage.GetTenant) → filter by tenant
      4. If admin role present and same tenant → skip owner filter
      5. Else → filter by owner (identity.Subject)
      6. If resource not found after filtering → return ErrNotFound (404)
      7. If ownership denied → slog.Debug with subject, resource ID, operation
```
