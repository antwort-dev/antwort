= Prometheus Metrics Reference
:description: Complete reference for all Prometheus metrics exposed by Antwort.

Antwort exposes Prometheus metrics at the `/metrics` endpoint (configurable via `observability.metrics.path`).
All metrics use the `antwort_` prefix.
Enable metrics with `observability.metrics.enabled: true` in the configuration.

== HTTP Layer (Spec 013)

[cols="3,1,2,3"]
|===
| Metric | Type | Labels | Description

| `antwort_requests_total`
| Counter
| `method`, `status`, `model`
| Total HTTP requests by method, status class (2xx/4xx/5xx), and model.

| `antwort_request_duration_seconds`
| Histogram
| `method`, `model`
| HTTP request duration.

| `antwort_streaming_connections_active`
| Gauge
| (none)
| Current number of active SSE streaming connections.
|===

== Responses Layer (Spec 046)

[cols="3,1,2,3"]
|===
| Metric | Type | Labels | Description

| `antwort_responses_total`
| Counter
| `model`, `status`, `mode`
| Total responses by model, completion status (completed/failed/cancelled/incomplete), and mode (sync/streaming/background).

| `antwort_responses_duration_seconds`
| Histogram
| `model`, `mode`
| End-to-end response duration by model and delivery mode.

| `antwort_responses_active`
| Gauge
| `mode`
| Currently in-flight responses by mode.

| `antwort_responses_chained_total`
| Counter
| `model`
| Responses using `previous_response_id` for conversation chaining.

| `antwort_responses_tokens_total`
| Counter
| `model`, `type`
| Cumulative token usage per response (input/output).
This counts total tokens for the entire response, including all agentic loop turns.
|===

== Histogram Bucket Configurations

All duration histograms use LLM-tuned buckets unless noted:

[source]
----
LLMBuckets:              0.1, 0.5, 1, 2, 5, 10, 30, 60, 120 (seconds)
ConversationDepthBuckets: 1, 2, 5, 10, 20, 50 (item count)
TokenBuckets:            1, 4, 16, 64, 256, 1024, 4096, 16384 (token count)
----

== Label Cardinality

Some labels accept unbounded values.
Operators should use Prometheus relabeling to control cardinality in production.

[cols="2,1,3"]
|===
| Label | Cardinality | Notes

| `model`
| Unbounded
| Comes from the request `model` field.
Use `metric_relabel_configs` to group or drop high-cardinality model names.

| `tool_name`
| Medium (3-20)
| Built-in tools are fixed. MCP tools add cardinality per configured server.

| `store_id`
| Medium
| One series per vector store.
Grows with store count.

| `worker_id`
| Low-Medium
| One series per worker process.
New ID on each restart.
Consider using stable identifiers (pod name) in Kubernetes.

| `content_type`
| Low-Medium
| MIME types from uploaded files.

| `status`, `mode`, `operation`, `result`, `backend`
| Fixed (2-4 values)
| Bounded enums, no cardinality risk.
|===

== Example Prometheus Relabeling

To drop high-cardinality model labels:

[source,yaml]
----
metric_relabel_configs:
  - source_labels: [__name__, model]
    regex: "antwort_responses_total;(.+)"
    target_label: model
    replacement: "${1}"
    action: replace
----
